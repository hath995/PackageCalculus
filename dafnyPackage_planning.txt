❯ what sort of features in this calculus would be useful for a package manager  
  for Dafny?                                                                    

● Fun question — and Dafny is actually an unusually interesting fit for this
  calculus, because verification changes which extensions matter. Here's my
  read, mapping the paper's axes onto what a Dafny package manager would
  actually face.

  Where Dafny sits on the paper's spectrum (§3.3)

  Dafny has the OCaml problem, but worse. In OCaml, any API change is
  potentially breaking; in Dafny, even proof-irrelevant-looking changes break
  downstream verification — a changed function body alters what callers see
  through revealed definitions, triggers shift, resource limits get exceeded.
  Semantic versioning can't honestly promise "minor = compatible" when
  compatibility includes "your proofs still go through." So:

  - MVS is out (it relies on semver discipline Dafny can't have), and
  upper-bound-free constraints are risky.
  - That pushes Dafny to the opam end: version uniqueness, full version formulae
  (§3.2), NP-complete resolution, a curated registry with reverse-dependency 
  re-verification on new releases — opam's model, except the reverse-dependency
  check is dafny verify, which is a much stronger and more meaningful
  compatibility check than OCaml's type-level one. The calculus's core + version
  formula layer is exactly this, and both are mechanised here.
  - Concurrent versions (§4.2) and peer deps (§4.3) are the extensions to skip.
  Dafny's module namespace is flat per program and there's no name-mangling
  story for linking two versions of Std into one verification run. Cargo's trick
  doesn't transfer.

  The extensions that map surprisingly well

  Virtual packages (§4.7) ≈ abstract modules. Dafny has a language-level notion
  the other ecosystems fake: a package can depend on an abstract module 
  signature, and any package providing a concrete refinement satisfies it.
  That's literally the provides relation — depend on "ordered-map-interface"; 
  provided by red-black-tree-pkg or avl-pkg. Also the toolchain itself: a
  dependency on smt-solver provided by z3 or cvc5. This is probably the
  highest-value "exotic" extension for Dafny.

  Variable formulae (§4.6) for compilation targets and the toolchain. A Dafny
  package compiles to C#, Java, Go, JS, Python — :extern dependencies are
  inherently target-conditional. That's opam's os-distribution pattern verbatim:
  ¬(target = java) ∨ java-runtime-shim. Global variables for dafny-version and
  solver-version matter more than in any mainstream ecosystem, because proofs
  are brittle against both — the toolchain version belongs inside resolution,
  not outside it. Local variables give you with-test / with-docs dependency
  groups for free.

  Features (§4.4) for trust levels. A genuinely Dafny-specific use: feature
  flags like axiom-free or assume-free selecting the fully-verified subset of a
  library versus the portion that relies on {:axiom}/:extern trust. Feature
  unification has the right semantics — if anything in your closure demands the
  verified-only build of a dependency, everyone gets it.

  Singular dependencies (§4.9) + the build graph (§4.8) for reproducible 
  verification. Lockfiles aren't just nice for Dafny, they're what makes a
  verification cache sound: "this module verified" is only meaningful relative
  to the exact text of every dependee and the exact toolchain. A lockfile is a
  singular-dependency instance (the embedding is proved here), and Nix-style
  verification-cache substitution — skip re-verifying dependencies whose
  (content hash, toolchain) pair is already certified — is the analogue of
  binary caching. The build graph is your verification order.

  Conflicts (§4.1), modestly: two packages exporting the same top-level module
  name, or demanding incompatible global flags (--unicode-char, type-system
  mode), genuinely can't coexist.

  The self-hosting punchline

  A Dafny package manager should be written in Dafny — and this repo is most of
  its verified core. The validity checker is compiled and executable; the
  reductions are proven compilation passes, so the resolver only ever needs to
  solve core instances; and Appendix C's SAT encoding is verified
  (EncodeSound/EncodeComplete), with Dafny already shipping Z3. The pipeline
  "parse manifest → lower extensions to core (proven) → encode to SAT (proven) →
  ask the Z3 you already bundle → decode (proven)" would make it the first
  package manager whose resolver is verified end-to-end against its own
  semantics.

  If you wanted to prototype that, the missing pieces are all at the edges, not
  the middle: a manifest format, a registry, and a content-hashing scheme for
  the verification cache — the formal core is sitting in src/.

❯ what if we had a package system that assigned a version to a new version      
  based on how compatible it is to the previous version. let's say that it only 
  adds function or definitions then it be fully compatible. but if a function   
  signature changed in types it would be Marked as incompatible or if the       
  function preconditions were stronger than before or if the post conditions    
  got weaker?                                                                   

● You've essentially re-invented behavioral subtyping (Liskov substitution)
  applied to releases — and Dafny is the one ecosystem where this idea actually
  closes, because the compatibility judgment is itself a verification condition.
  "Is pre_old ⇒ pre_new, and does post_new ⇒ post_old under pre_old?" — those
  are formulas you can hand to the verifier you already have. Elm does this for
  types (the compiler computes the semver bump for you); your proposal is "Elm,
  but for contracts." Your variance directions are exactly right, and they have
  a useful converse: a release that weakens preconditions or strengthens
  postconditions is also fully compatible, not just additions.

  How it lands in the calculus we mechanised:

  It rehabilitates the tractable end of §3.3 for Dafny. Earlier I argued Dafny
  must sit at the opam end — NP-complete resolution, no semver — because
  "compatible" can't be promised socially. Your scheme replaces the social
  promise with a theorem. The paper's approach (1) — restrict constraints to
  minimum bounds, get linear-time MVS-style resolution — is sound exactly when
  versions above the bound really are substitutable, which Go assumes and your
  checker would prove. And the granularity function g of §4.2 (already in the
  mechanisation) is precisely your compatibility class: g(v) = number of 
  proven-incompatible steps before v. Checking only against the immediate
  predecessor suffices, because your relation is transitive — implication chains
  compose — so one VC set per release certifies compatibility with the whole
  class.

  What "compatible" has to mean in Dafny, beyond your three rules. This is where
  the idea gets sharp teeth, because several Dafny-specific things silently
  belong to the spec surface:

  - Transparent function bodies. Dafny functions are revealed by default;
  callers' proofs can depend on the body, so a body change with identical
  contracts is still breaking — unless the checker proves extensional equality
  (∀x · f_old(x) == f_new(x), another VC) or the ecosystem adopts
  opaque-plus-contract discipline, with export sets making the spec surface
  explicit.
  - Datatypes are nearly frozen within a class. Adding a constructor breaks
  every exhaustive match and every case-analysis lemma downstream. Additions are
  not always compatible — that also covers names colliding with clients' import
  opened.
  - Frames are contracts too. A grown reads/modifies clause breaks callers'
  framing arguments; frames must shrink or stay equal, same as preconditions.
  - Lemmas are specs. Weakening a lemma's conclusion or strengthening its
  hypotheses is breaking by exactly your rule — lemmas are just ghost methods.
  - The trust base needs its own gate. A "compatible" release that adds an
  {:axiom}, assume, or :extern can make clients prove more than before —
  including, in the worst case, false. Pre/post variance doesn't see that; the
  checker should flag any growth of the unverified surface as at least a major
  bump.

  One honest caveat: spec-level compatibility guarantees clients remain
  logically verifiable, not that their proofs replay — triggers shift, resource
  limits get crossed. So the guarantee is "no client is wrong now," not "no
  client needs proof repair." Practically that means pairing auto-versioning
  with lockfiles for CI (the §4.9 singular instance) while letting dev builds
  float within a compatibility class, and the conservative failure mode is
  benign: when the verifier can't prove compatibility, you bump major — never
  unsound, just occasionally over-cautious.

  There's also a neat expressivity bonus: since classes are machine-checked, you
  could safely do what Go does — treat distinct compatibility classes as
  distinct package names — and then the Concurrent calculus (§4.2, fully proved
  here) applies with g = class, if Dafny ever grew a story for co-existing
  module instances.

  If you want to push this into the repo, the natural next artifact is the one
  piece of §3.3 we haven't mechanised: define min-bound dependencies over
  compatibility classes, implement the greedy MVS-style resolver, and prove it
  produces valid resolutions — making your versioning scheme's payoff
  (linear-time, lockfile-optional resolution) a verified theorem rather than a
  slogan. The substitutability side ("a class-compatible upgrade preserves
  client spec-validity") lives at the program-semantics level the paper
  deliberately doesn't model, but the resolution side is squarely provable with
  what we have. Happy to build it if you'd like.

  2. §3.3 mechanised — with two new findings

  Approach (1), minimum bounds (MVS): MvsVisited is a verified terminating
  worklist computing the least bound-closed visited set, MvsOf takes per-name
  maxima — Cox's actual algorithm. MvsValid proves the output is always a valid
  resolution ("greedy suffices, no backtracking"), MvsDeterministic proves the
  answer is a function of the bounds (the "lock files are unnecessary" claim),
  UpgradeToleratesBounds proves the "any version above the maximum of all
  minimum bounds" sentence, and MinVfEquiv/MinToCoreCorrect show min-bounds are
  the ≥-fragment of §3.2, reducing to the core via the already-proven Theorem
  3.2.7.

  Approach (2), no uniqueness: MultiResolve is a verified graph-traversal
  resolver that always succeeds, and MultiToConcurrentIdentity/ConcurrentToMulti

● Short answer: the logic is exactly as hard as ordinary Dafny verification —
  semi-decidable, usually automatic, occasionally needs a hand-written lemma —
  and the scheme can absorb that gracefully. The genuinely annoying part isn't
  comparing the predicates, it's getting both versions of them into one program.
  Let me break that down.

  The judgment itself is just an implication VC. forall args :: pre_old(args) 
  ==> pre_new(args) is precisely the kind of formula Dafny discharges all day.
  For the bulk of real contracts — bounds checks, null/validity conditions,
  arithmetic — these sit in decidable or well-behaved SMT fragments and verify
  instantly. Quantified specs bring the usual E-matching/trigger incompleteness,
  but that's the same incompleteness Dafny users already live with, and the
  failure mode is benign by design: "solver says unknown" → conservative major
  bump. You can lose precision, never soundness.

  There's a fast path that covers most of the work. Releases rarely touch most
  contracts. Diff first: alpha-equivalent contract ASTs are compatible with zero
  solver calls. Among the changed ones, the common edits are conjunct-level and
  syntactically free — dropping a conjunct from requires is weakening (A ∧ B ⇒ 
  A), adding a conjunct to ensures is strengthening. The solver only sees
  genuine semantic rewrites, and those are a small tail.

  The hard tail is real but has a native escape hatch. Implications between
  recursive predicates — sorted_v2(s) ⇒ sorted_v1(s) where both are recursively
  defined — typically need induction, which SMT won't find alone. Here's where
  Dafny is uniquely positioned: the checker can emit the unproved obligation as
  a lemma stub, and the library author proves it by hand — lemma 
  PreStillWeaker(x) requires pre_old(x) ensures pre_new(x) — in the language
  they already write proofs in. The compatibility claim becomes one more proof
  obligation shipped with the release. Elm can't offer that option; Dafny can.
  Undecidability degrades to "occasionally requires a manual lemma," which is
  just… Dafny.

  The actual engineering pain: two versions, one program. pre_old and pre_new
  come from different snapshots of the codebase and may call helper functions
  that themselves changed. You need to import both versions with symbols renamed
  apart (prefix everything from v1 with Old_, say) so the implication is even
  expressible — this is the relational-verification plumbing that tools like
  SymDiff (differential assertion checking) built for C. It's mechanical but
  it's the bulk of the tool. Three side-effects of that plumbing to watch:

  - Shared vocabulary. If a helper predicate both contracts mention changed
  meaning, the naive check silently compares against different definitions;
  renaming apart makes that explicit, but then trivial-looking implications stop
  being trivial (Old_Sorted(s) ⇒ Sorted(s) is now a real obligation, correctly
  so).
  - Abstraction changes. If the spec is stated over an abstract model (Repr, a
  ghost map) whose representation changed, you need a coupling invariant
  relating old model to new — full data-refinement territory, genuinely hard in
  general. Pragmatic answer: if the exported model type changed, that's a
  signature change, i.e. major bump by your own rule; only compare contracts
  over an unchanged spec vocabulary.
  - Heap specs. Frames compare easily when syntactic (modifies set inclusion),
  less easily when dynamic (reads Repr); two-state postconditions need the
  implication checked under pre_old and the old frame.

  Worth noting: Dafny's module refinement already enforces a conservative cousin
  of this (a refining module can add ensures but, IIRC, can't touch requires) —
  so the covariant half has language precedent; the full contravariant check is
  what the external tool adds.

  So: not difficult in principle, mostly cheap in practice, occasionally
  lemma-shaped at exactly the points where Dafny authors are already comfortable
  writing lemmas — and the real implementation cost is version-pairing
  plumbing, not logic.

  predecessor makes it self-contained. A few concrete things you'd hit building
  it, including one non-obvious wrinkle that shapes the whole generator:

  Contracts aren't first-class, so the generator must splice expressions. Dafny
  has no requires-of(f) operator — you can't write a lemma that abstractly
  references "f's precondition." The generated file has to copy the contract 
  expressions out of both versions' ASTs, with the old version's symbols
  qualified apart. So the artifact looks like:

  lemma Find_PreWeakened(s: seq<int>, key: int)
    requires Old.Lib.SortedAsc(s) && |s| > 0     // v1's requires, Old-qualified
    ensures Lib.SortedAsc(s)                      // v2's requires
  {}

  lemma Find_PostStrengthened(s: seq<int>, key: int, idx: int)
    requires Old.Lib.SortedAsc(s) && |s| > 0      // under the OLD precondition
    requires PostV2(s, key, idx)                  // v2's ensures, spliced
    ensures  PostV1(s, key, idx)                  // v1's ensures, Old-qualified
  {}

  That means the tool needs Dafny's parser (the Microsoft.Dafny API, or dafny 
  --print of a resolved program as a poor-man's extraction) — string-level
  diffing won't survive qualified names.

  You may not even need to rename modules — wrap them. Rather than rewriting
  module Foo to module V1_Foo and chasing every internal reference, concatenate
  the old release's sources inside one module Old { … }. Module references
  resolve lexically, so the old code is self-contained under Old.*, while
  references to the shared standard library escape to the common Std — which is
  exactly what you want, since both versions' contracts should speak a shared
  vocabulary for Std. The fiddly residue is include directives (file-level, so
  you concatenate first) and :extern attributes (irrelevant for
  verification-only runs).

  The failure modes compose beautifully. Signature changed in types? The
  generated lemma doesn't even resolve — type error → major bump, no solver
  involved. Contract textually unchanged? Skip generation entirely. Conjunct
  dropped from requires? The lemma is A ∧ B ⇒ A, discharged instantly with the
  empty body. Only genuine semantic rewrites reach the solver, and dafny verify 
  compat.dfy's exit code is the bump decision.

  Leave room for manual proofs without clobbering them. Dafny won't let you
  separate a lemma's declaration from its body, so use a convention: the
  generator emits {} bodies by default, but if the author's hand-maintained
  compat-proofs.dfy defines Find_PreWeakened_Manual with the matching signature,
  the generator emits { Find_PreWeakened_Manual(s, key); } instead.
  Regeneration stays idempotent; inductive cases get human help; everything
  remains assumption-free.

  Two cases need more than the simple template. Methods with heap effects: the
  post-comparison needs the old-state/new-state structure (old() expressions,
  frames), so the harness is a generated method over a havocked frame rather
  than a lemma — SymDiff-style; I'd ship v1 of the tool covering functions,
  lemmas, and ghost predicates (where Dafny libraries keep most of their spec
  weight) and add the heap harness second. And transparent function bodies: for
  exported non-opaque functions whose body changed, generate ensures 
  Old.Lib.F(x) == Lib.F(x) — body-equivalence as just another compat lemma.

  The registry closes the trust loop. Don't trust the publisher's bump: the
  compat file ships with the release and the registry re-verifies it on upload —
  the compatibility claim becomes an auditable, machine-checked artifact
  attached to the version, not metadata someone asserted. And that's the point
  where it plugs into what we proved yesterday-in-this-repo: the registry
  assigns the compatibility class, classes are the granularity function, and the
  §3.3 results (MvsValid, MvsDeterministic) give you a verified linear-time
  resolver over min-bounds within a class — with the fresh-root caveat already

● Two questions — let me take the versioning one first because it has a crisp
  answer that I think improves your scheme, then the worth-it question.

  "Compatible by lemma": minor or major?

  First, a distinction that matters: whose lemma?

  Publisher-side lemma (our _Manual hook): unambiguously doesn't affect the
  bump. The bump should track whether the implication is true, not whether Z3
  found the proof unassisted. Once IndexOf_PreWeakened is proved — by whatever
  means — the semantic guarantee to clients is identical.

  Client-side lemma — your actual question — and here's the subtlety:
  publisher-proven compatibility does not guarantee client proofs replay
  mechanically. Concretely, in our compat/ example: a client whose loop
  invariant establishes adjacent-sortedness inline will, after the upgrade, face
  a call site demanding pairwise-sortedness. The implication is true (the
  publisher proved it!), but the client's verifier needs the same induction at
  the call site. Semantically nothing broke; mechanically the client must add
  one lemma call. Notice the diagnostic: if the publisher's obligation needed a 
  manual proof, client call sites are likely to need the same bridge.

  So the honest taxonomy is three-valued, and — pleasingly — semver already has
  three digits:

  - Patch: every obligation discharged with an empty body (unchanged contracts,
  dropped conjuncts, trivial implications). Client contexts that entailed the
  old contract entail the new one by the same shallow reasoning — no edits
  expected.
  - Minor = your "compatible by lemma": all obligations proved, but some needed
  _Manual. Semantically safe — no client is wrong — but clients may need proof
  repair. Two policies should attach to this tier: the publisher must export the
  bridging lemmas (the compat file tells you exactly which ones — they're the
  _Manual entries), so the client's fix is a one-line lemma call; and tooling
  can suggest it, since the failing client obligation matches a shipped bridge
  by shape.
  - Major: some implication is false (witness exists) or the obligation doesn't
  resolve (signature change).

  This has a nice precedent — Rust's RFC 1105 explicitly allows "minor changes
  that may require annotations" (new trait impls can break inference) — and it's
  computable from the artifact we already built: the {}-body vs _Manual-body
  distinction in the generated file is literally the patch/minor classifier.
  It's a proxy, not a theorem about client sites (the publisher's check is one
  fixed formula; client contexts vary), so I'd present it as: patch = "no new
  proof obligations introduced," minor = "safe, bridging lemmas shipped," major
  = "callers can be wrong." Resolution policy then keys off it naturally:
  auto-float patch everywhere, float minor in dev / lock in CI, never float
  major — and the §3.3 min-bound machinery we verified applies within whichever
  tier you let float.

  Would it be worth building?

  The strongest argument: the checker is valuable before any package manager 
  exists. Proof-library upgrades are the notorious cost center of every
  verification ecosystem — mathlib handles it by being a monorepo with
  continuous breakage-fixing; Coq handles it by pinning everything in the
  Platform; Dafny users today mostly vendor and freeze, which is the same
  admission. A standalone "spec-diff" CI bot — runs on every PR of a library
  repo, comments "this strengthens IndexOf's precondition: breaking; this
  strengthens Min's postcondition: safe" — needs no registry, no resolver, no
  ecosystem buy-in, and would already pay for itself in any industrial Dafny
  codebase. That's the incremental path: CI bot → auto-versioned registry → MVS
  resolution, each stage useful alone.

  Would it make libraries easier to use? I think the bigger effect is making
  them safe to depend on at all. Today the rational Dafny strategy is to copy
  library source into your repo precisely because an upgrade might cost you a
  week of proof repair with no warning. Machine-checked compatibility classes
  convert "upgrading is gambling" into "patch upgrades are free, minor upgrades
  come with the exact lemmas you'll need, major upgrades are honestly labeled."
  Secondary benefits compound: the compat file doubles as a precise
  spec-changelog for review; the scheme pressures library authors toward
  opaque-function-plus-contract discipline and explicit export sets (since
  transparent bodies are effectively frozen), which improves library design
  independently.

  Honest costs: the generator is real engineering (a Microsoft.Dafny AST pass,
  two-version plumbing, the heap-method harness, traits — months, plus tracking
  Dafny releases); conservative false-majors will happen when authors don't

❯ So lets say we have one predicate sorted which is defined one way in v1 and a 
  different way in v2. The lemma would require the old version to remain in     
  the code base?                                                                

● No — the old declaration doesn't have to survive anywhere in the shipped code.
  The old definition shows up in two places, and both have a way around keeping
  it:

  1. The compatibility check itself. Here both definitions genuinely must
  coexist — that's exactly what the module V1 { … } wrapping is for. But it's a
  transient verification environment, not the codebase: the registry
  materialises v1 from its own archive, wraps it, verifies CompatV1V2.dfy, and
  the v2 package ships only v2 sources. Since the registry keeps every release
  forever, the check is reproducible on demand without v2 carrying any v1 text.
  (Same answer as "where does the old binary live for an ABI-diff tool" — in the
  artifact store, not in your tree.)

  2. The client-facing bridge lemma — the subtler case. The naive bridge
  V1.Sorted(s) ⇒ V2.Sorted(s) would indeed force the old predicate to stay
  exported. But that's the wrong statement to ship. Think about what the client
  actually has: their old proof established the old definition's content — say,
  adjacent-sortedness from a loop invariant — not the old name. So the useful
  bridge is stated entirely in v2's vocabulary, with the old body inlined as a
  plain formula:

  // shipped in v2, no reference to any v1 declaration
  lemma SortedFromAdjacent(s: seq<int>)
    requires forall i | 0 < i < |s| :: s[i-1] <= s[i]   // v1's body, as a
  formula
    ensures Sorted(s)                                    // v2's predicate

  The old definition survives as a hypothesis formula inside one lemma — which
  the generator can produce mechanically, since it's holding both ASTs anyway —
  but no Sorted_v1 name pollutes the API. A publisher who wants more
  discoverability can optionally keep a deprecated alias predicate 
  SortedAdjacent plus the bridge; that's taste, not necessity.

  One sharpening your example exposes, though. A changed predicate definition is
  a stronger event than a changed precondition, because exported predicates
  appear in client code at both polarities — someone requires Sorted(s), someone
  else ensures it, someone branches on !Sorted(s). A one-directional
  implication can't cover all of them. So the right generated obligation for an
  exported transparent predicate whose body changed is equivalence, V1.Sorted(s)
  <==> V2.Sorted(s) — which our adjacent/pairwise example happens to satisfy
  (pairwise ⇒ adjacent is the trivial direction; adjacent ⇒ pairwise is the
  induction). Our CompatV1V2.dfy only emitted the one direction that IndexOf's
  precondition needed; a complete ruleset would emit the iff for Sorted itself.
  And the corollary is a clean design rule for publishers: if you want to
  genuinely change what a predicate means — not just rephrase it — don't
  redefine the name; introduce a new predicate and migrate your own contracts to
  it. The old name's removal (or its broken equivalence obligation) then
  registers as the major bump it really is, and "redefinition" stops being a way
  to silently shift meaning under every client simultaneously.

  So: check environment holds both versions transiently; shipped library holds
  only v2 plus bridge lemmas phrased in v2's own vocabulary; and predicate
  redefinitions get held to the equivalence standard, with "new meaning = new
  name" as the escape valve. Happy to add the Sorted equivalence obligation and
  the v2-vocabulary bridge to the compat/ example if you want the convention
  pinned down for that case too.

● Both dimensions slot into machinery we've already mechanised — and pleasingly,
  each one lands on a different piece of it.

  The Dafny version: model the toolchain as a package

  Make dafny (and z3/cvc5) ordinary packages in the repository with versions
  4.10, 4.11, 4.12, and let libraries depend on them with version formulae
  (§3.2). Two things fall out for free:

  - Toolchain coherence is just version uniqueness. A program is verified by one
  Dafny invocation, so every library in a resolution must agree on one
  toolchain version — which is literally Definition 3.1.3(c) applied to the
  dafny package. No new mechanism; the core calculus's most basic rule does the
  work. And resolution gets to trade off jointly: if libA@2 needs dafny ≥ 4.11
  but libB's latest only supports 4.10, the solver can fall back to libA@1 —
  toolchain choice and library choice are one constraint problem, which is
  exactly why Dafny lives at the NP-complete/SAT end of §3.3.
  - Support ranges should be measured, not declared. The contract-derived
  versioning story doesn't apply here — you can't prove dafny 4.11 ⊒ 4.10 by
  lemma; proof brittleness across toolchain releases is empirical. So the
  registry runs each library release against its Dafny matrix and records which
  versions verify, generating the dafny constraint from CI fact rather than
  author optimism. Note the support set can be non-contiguous (verifies on 4.10
  and 4.12 but a 4.11 regression breaks it) — which is precisely why §3.2's
  general version formulae matter over simple ranges, and that reduction is
  proven (VfReductionCorrect).

  Options and toggles: the §4.6 variable formulae, used for real

  The key is to sort options into three buckets, because they want three
  different treatments:

  1. Semantics-changing, globally coherent — --unicode-char,
  --type-system-refresh, --quantifier-syntax, --standard-libraries. These change
  what source means, and one invocation has one setting, so the whole
  resolution must agree. That is exactly the paper's global variable (§2.2.12's
  os-distribution, mechanised in §4.6): manifests contribute constraints like
  unicode-char = true or ¬(type-system = legacy) — negative constraints
  included, since our Tseitin-backed reduction handles negated comparisons.
  Three consequences, all already proved:
  - coherence is enforced because the reduction turns each option into a
  variable package whose version is its value — version uniqueness again
  (VarReductionSound/Complete);
  - the resolver can solve for option values: if A requires unicode-char = true
  and B is indifferent, resolution picks true; if B requires false, you get a
  comprehensible UNSAT instead of a mid-build mystery — the paper's "variables
  can themselves be solved for," operational;
  - a library can make dependencies conditional on options (¬(standard-libraries
  = true) ∨ my-std-shim), the opam filtered-dependency pattern we tested in
  TestVariableFormulae.

  2. Verification-effort, locally scoped — --resource-limit,
  --isolate-assertions, manual trigger style. These affect whether this
  package's proofs go through, not what neighbors' code means. They're §4.6's
  local variables (the with-test pattern) or plain per-package metadata; they
  impose nothing on the rest of the closure and shouldn't enter the global
  constraint problem.

  3. Certificate-relevant — solver identity and version, rlimits actually used,
  flags like --allow-axioms. These don't constrain resolution; they qualify what
  "this package verified" means. They belong in the verification-cache key and
  the lockfile: a cached certificate is (content hash, dafny version, solver 
  version, semantic-option assignment), and substitution is only sound on exact
  match (with rare monotone exceptions like "verified at rlimit X ⟹ fine at ≥
  X").

  The pleasing punchline

  Definition 4.6.2's resolutions are pairs (R, α) — packages plus a variable 
  assignment. So "the lockfile must record the option settings" isn't a bolt-on
  engineering decision: in the mechanised semantics, the assignment is already
  half of the resolution object. A Dafny lockfile is literally an (R, α) of the
  Variable Formula Calculus, with R pinning library and toolchain packages and α
  pinning the toggle values — and AsgOf/AsgPkgs are the proven round-trip
  between that assignment and the solver's view of it.
